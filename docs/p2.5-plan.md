# P2.5 — Onboarding & project lifecycle

> Goal: a freshly logged-in user can sign in, create their first organization
> and project from the web UI, see the new project's DSN, and reach the issues
> list for it — without anyone hand-running SQL.
>
> Discovered as a gap between P1 ("ingest works against a pre-existing DSN")
> and P2 ("user can browse issues that already exist"): no milestone owned the
> _create_ side of the lifecycle, so `OnboardingPage.tsx`'s submit handler was
> a `console.warn('TODO', ...)` and `ProjectsPage.tsx` rendered a hard-coded
> placeholder array.

## Architecture decisions

- **User mirror sync.** Keycloak is the source of truth for identity, but
  `org_members.user_id` has an FK to `users(id)`. We UPSERT the `users` row
  on the call to `POST /orgs` (and any other write that needs membership)
  using `sub`, `email`, and `preferred_username` from the JWT. Lazy is good
  enough for now; a request filter can replace it in P4 if every write
  starts needing it.
- **Slug derivation.** Server derives `slug` from `name` (lowercase ASCII +
  hyphens). Collisions get a `-2`, `-3`, … suffix in the same transaction
  via a SELECT-then-INSERT with a unique-violation retry. Slugs are not
  user-editable in P2.5; rename can come later if needed.
- **Guard scoping.** `OrgAccessGuard` is registered at `/api/v1/orgs/*/**`
  rather than `/api/v1/orgs/**` so `POST /api/v1/orgs` (no `{orgId}` yet)
  and `GET /api/v1/orgs` (list-mine) bypass the guard but stay
  authenticated. Inside the create-org service, we _do not_ set `OrgContext`
  — the writes go through tables that don't have RLS (`organizations`,
  `users`, `org_members`).
- **DSN host in response.** `POST /projects/{id}/keys` returns the DSN
  string `argus://<dsn_public>@<ingestHost>/api/<projectId>` exactly once.
  The api reads `argus.ingest.public-host` from properties (defaults to
  `http://localhost:8080`); production overrides via env. After the modal
  closes, only the truncated public id is shown again — by design.
- **Onboarding atomicity.** Web fires three sequential mutations
  (org → project → key). If the project step fails after org succeeded,
  the user lands on `/orgs/{slug}/projects` with an empty state and a
  retry CTA. We don't wrap these in a single backend transaction; the
  failure mode is rare and a multi-resource POST clouds the URL design.

## Carry-forwards from earlier phases

- Hex layout: `domain` / `application` (ports, services) / `adapter/in/web`
  / `adapter/out/postgres`. Same pattern as `alerts/` (P3 #1–#2).
- Tests: pure unit (validation, slug derivation), Testcontainers
  integration (real Flyway-migrated Postgres for write paths), controller
  test with `@WebMvcTest` for routing.
- `OpenApiContractTest` snapshot regenerates with `ARGUS_OPENAPI_WRITE=true`.
- Frontend: TanStack Query mutations, no Zustand (auth store is the
  exception). Errors are surfaced via `@mantine/notifications`.

## Out of scope (revisit later)

- Org / project rename, transfer of ownership, deletion.
- Invitations & multi-user membership (today: only the creator is a member).
- Slug edits.
- DSN rotation / revocation UI (the schema supports `active=false` already;
  the route is added but no web button until needed).
- Onboarding "skip — I have a DSN already" path.
